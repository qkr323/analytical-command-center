  1. 1
    How it works
    Futu HK (no_reply@stmt.futuhk.com) and SoFi HK (no-reply@sofi.hk) send daily statement emails with PDF attachments to sam.sanghyunpark@gmail.com.

    A Gmail filter auto-forwards these to qkr323@gmail.com. The script sync_from_email.py connects to Gmail API, downloads the PDFs, and uploads them to the backend.
  2. 2
    Gmail credentials (one-time setup)
    Credentials are stored at:
    backend/gmail_credentials.json — OAuth2 client secret (from Google Cloud Console, project "Portfolio Tracking")
    backend/gmail_token.json — auto-saved access/refresh token (never expires unless revoked)

    Both files are gitignored. To re-create if lost: Google Cloud Console → "Portfolio Tracking" project → APIs & Services → Credentials → "Portfolio Sync" client → Download JSON.
  3. 3
    What the script does
    python sync_from_email.py
    1. Searches Gmail for emails from Futu/SoFi without the "portfolio-synced" label
    2. Downloads the PDF attachment from each email
    3. POSTs to http://localhost:8001/upload/statement with the correct broker and account_id
    4. Labels the email "portfolio-synced" so it's never re-processed

    Account IDs: Futu=2, SoFi=4
  4. 4
    Futu PDF parser
    The Futu daily statement PDF has 10+ pages. Positions are in the "Ending Assets Overview — Stocks" section (end-of-day prices). Some ETF names wrap across two lines (e.g. TLT, EDV, KWEB, LIT) — the parser handles this with a line-joining step before applying regex.

    Parser: backend/services/parsers/futu.py
    Accounts for: all 12 positions (HK stocks, US stocks/ETFs, CNH ETF)
The gmail_token.json contains a refresh token and doesn't expire. Future runs are fully silent — no browser window needed.
⚠️ This method is no longer the primary workflow. Futu is now synced daily via Gmail PDF statements (see "Email Sync Setup" tab). This tab is kept for reference and as a fallback.
  1. 1
    Why we originally used Futu OpenD
    Futu does not have a public internet API. Instead, they provide a local program called Futu OpenD that runs only on your Mac at 127.0.0.1:11111 and exposes a Python API (the futu-api package).

    The backend connects via services/brokers/futu_opend.py. When you trigger a sync, the local backend calls OpenD, which connects to Futu's servers, fetches positions, and writes them to the Railway DB.

    This worked, but required OpenD running, a local backend running, and terminal commands every time. The website sync button always failed (Railway's 127.0.0.1 is Railway's own container, not your Mac).
  2. 2
    Fallback: sync via OpenD if needed
    Launch Futu OpenD from Applications. Wait until the log shows "The necessary data is ready".
    open -a FutuOpenD
  3. 3
    Start local backend + run OpenD sync
    # Terminal 1 — start engine cd /Users/sampark/projects/portfolio-tracker/backend source ../.venv/bin/activate python -m uvicorn main:app --port 8001 # Terminal 2 (Cmd+T) API_KEY="9f5c7eb34194861330d08176b09050f4a33bf334e1e70915e2415bb4adc9c8bb" curl -X POST "http://localhost:8001/sync/futu" -H "X-API-Key: $API_KEY"
  4. 4
    Why we switched to Gmail PDF sync
    Futu HK sends a daily statement email to sam.sanghyunpark@gmail.com each morning with a PDF containing all positions and end-of-day prices. By reading this email via Gmail API and uploading the PDF to /upload/statement, we get the same data without needing OpenD running.

    The Futu PDF parser (services/parsers/futu.py) was rewritten to handle the daily statement format, including a line-join step for ETF names that wrap across lines (TLT, EDV, KWEB, LIT).
protobuf note: google-api-python-client requires protobuf 7.x but futu-api requires 3.x. Both are installed — the email sync works fine, but if you use the OpenD fallback and hit protobuf errors, pin protobuf back to 3.x temporarily.
  1. 1
    Why Binance must be synced from your Mac
    Binance.com is geoblocked on Railway's servers — Railway runs on US/EU cloud infrastructure where Binance.com is restricted. When you click the Binance button on the website, Railway tries to call Binance.com and gets a 451 Unavailable For Legal Reasons error.

    Your Mac has no such restriction. So the fix is simple: run the sync from your Mac terminal through the local backend, which calls Binance.com directly from your home/office network.
  2. 2
    Start the local backend + sync
    # Terminal 1 — start engine cd /Users/sampark/projects/portfolio-tracker/backend source ../.venv/bin/activate python -m uvicorn main:app --port 8001 # Terminal 2 (Cmd+T) — run sync + prices API_KEY="9f5c7eb34194861330d08176b09050f4a33bf334e1e70915e2415bb4adc9c8bb" curl -X POST "http://localhost:8001/sync/binance" -H "X-API-Key: $API_KEY" curl -X POST "http://localhost:8001/sync/prices" -H "X-API-Key: $API_KEY"
  3. 3
    API key requirements
    Your Binance API key must have Read Info permission. It does NOT need trading permissions. The key and secret live in backend/.env and Railway environment variables — never in GitHub.
The Binance sync button on the website will show a friendly message reminding you to run it from your Mac terminal instead.
  1. !

  2. !

  4. !
