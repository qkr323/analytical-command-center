WHAT IS THIS?
-------------
A personal portfolio management web app that aggregates all your brokerage accounts
into one dashboard with compliance monitoring, live prices, and allocation charts.

Live dashboard: https://analytical-command-center.vercel.app


BROKERS CONNECTED
-----------------
  IBKR       — Flex Query API (automatic, no local app needed)
  Futu       — Daily PDF statement via Gmail (automatic via sync_from_email.py)
               Fallback: Futu OpenD local daemon (see HISTORY section below)
  Binance    — REST API via local backend (geoblocked on Railway — must run from Mac)
  SoFi       — Daily/monthly PDF statement via Gmail (automatic via sync_from_email.py)
  OSL        — PDF upload only (manual, monthly)
  Hang Seng  — PDF upload only (manual, monthly — download from Hang Seng online banking)


FRONTEND PAGES (6 navigation tabs)
------------------------------------
  Dashboard    — Portfolio summary: total NAV, allocation charts (by broker + asset type), positions table
  Positions    — Full position list with currency, native value, HKD equivalent, unrealized P&L
  Transactions — Transaction history with filters (broker, type, currency, ticker, date range); paginated
  History      — Historical portfolio value chart (area chart + by-broker + by-asset-type); NOT P&L
               — Now uses 30-day position snapshot rebuild: eliminates NAV dips on days without sync data
               — Positions are forward-filled with historical prices from yfinance + CoinGecko
  P&L          — Estimated trading P&L: realized, unrealized, dividend income, fees; data quality badges
  Currency     — Currency exposure breakdown (assets vs cash vs total, per currency, pie chart)
  Retirement   — AI Insights: retirement planning analysis, risk assessment, allocation advice

30-DAY POSITION HISTORY REBUILD
---------------------------------
  The History page chart would show artificial NAV dips on days when no sync data was received.
  This is now fixed with automatic position snapshot reconstruction:

  HOW IT WORKS:
    1. On every IBKR/PDF sync, the system reconstructs the last 30 days of daily position snapshots
    2. Deletes old snapshots for the rebuild window
    3. Loads opening positions (from before the window) with historical prices
    4. Replays all transactions during the window (buys, sells, transfers, dividends, fees)
    5. For each day: forward-fills positions with historical prices from yfinance (stocks/ETFs) and CoinGecko (crypto)
    6. Converts all prices to HKD using historical FX rates from frankfurter.app
    7. Creates clean daily snapshots for the entire 30-day window

  RESULT:
    - Portfolio NAV chart now shows smooth curves with no artificial dips
    - Historical prices used for days without broker sync data (weekend/holiday/no transaction days)
    - Accurate position-weighted NAV even on days the broker didn't report


P&L ENGINE (average-cost method)
----------------------------------
  The P&L engine calculates realized and unrealized trading P&L per position group (account_id, asset_id, currency).
  It is NOT a true investment return — it does not account for deposits, withdrawals, or capital timing.

  METHODOLOGY:
    - Dual running averages: avg_cost_local (always) + avg_cost_hkd (when FX available at import time)
    - HKD chain integrity: if any buy lacks gross_amount_hkd, avg_cost_hkd is broken for all subsequent sells
    - Partial history: sells with no cost basis history are excluded from P&L totals (exclude_from_pnl_totals=true)
    - Fees are included in cost basis (fees raise the average cost per unit)
    - Grouping key: (account_id, asset_id, currency) — same asset in different currencies tracked separately

  DATA QUALITY LEVELS (per broker):
    IBKR        — high (complete: trades, dividends, fees, deposits, withdrawals)
    Futu        — partial (trades only, history from Apr 2026)
    SoFi        — partial (trades only)
    Hang Seng   — partial (trades only, monthly gaps)
    OSL         — partial (trades only)
    Binance     — experimental (trades only, 90-day API window warning)

  NEW TRANSACTION COLUMNS (added 2026-04-25):
    realized_pnl_local, realized_pnl_hkd, cost_basis_local, cost_basis_hkd,
    avg_cost_per_unit_local, avg_cost_per_unit_hkd, cost_basis_method,
    calculation_version, data_quality_flag, pnl_calculated_at, exclude_from_pnl_totals

  ONE-TIME SETUP (must be run once against Railway DB):
    1. python -m scripts.migrate_pnl_columns      (adds 11 new columns — safe to re-run)
    2. POST /pnl/recalculate                        (replays all transaction history)

  P&L is automatically recalculated when new trades are imported (IBKR sync or PDF upload).
  Manual recalculation: "Recalculate P&L" button on the P&L page, or curl command in SYNC_INSTRUCTIONS.txt.


ACCRUAL TRACKING (IBKR Flex)
-----------------------------
  Interest and dividend accruals are now parsed from IBKR Flex reports and stored separately.
  This allows you to see earned but not-yet-received income per currency and per asset.

  INTEREST ACCRUALS:
    - Tracked per account and currency
    - Captured when opening IBKR Flex: set query to "Last 30 Days" and enable "Interest Accruals" in the form
    - Fields: starting_balance, interest_accrued, accrual_reversal, fx_translation, ending_balance

  DIVIDEND ACCRUALS:
    - Tracked per account, asset, and symbol
    - Captured when opening IBKR Flex: enable "Open Dividend Accruals" in the query form
    - Fields: quantity, ex_date, pay_date, gross_rate, gross_amount, tax, fee, net_amount

  HOW TO ENABLE:
    1. Go to https://clientportal.interactivebrokers.com → Flex Queries → Reports
    2. Click "edit" on the Portfolio query
    3. Set period to "Last 30 Days" (important for history rebuild)
    4. Scroll to "Interest Accruals" section → check the "Include Interest Accruals" box
    5. Scroll to "Dividend Accruals" section → check the "Include Open Dividend Accruals" box
    6. Save and run the query
    7. The next IBKR sync will automatically import these accruals

  NOTE: Accruals are useful for understanding cash flow timing and expected income.
  They do NOT affect the P&L calculation (only realized trades and dividends do).


KNOWN QUIRKS & FIXES
--------------------
  - HK stock prices sometimes fail in yfinance (ticker format issues) — prices show last known value
  - Futu daily PDF statements have long ETF names that wrap across lines — the parser handles this
    by detecting unclosed parentheses and joining with the next line before applying regex
  - Python 3.14 is too new — always use the .venv (Python 3.12) for the local backend
  - Hang Seng statement date format is DDMMMYYYY (e.g. 27MAR2026) — parsed by HangSengParser
  - Hang Seng ETFs 2800 (Tracker Fund) and 3416 (HSCEI ETF) are approved as broad index ETFs
  - protobuf conflict: google-api-python-client requires protobuf 7.x but futu-api requires 3.x.
    Both are installed in .venv — sync_from_email.py works fine; Futu OpenD sync may break if used.
  - Binance is geoblocked on Railway (HTTP 451) — must sync from Mac terminal via local backend
  - P&L columns require a one-time migration: run python -m scripts.migrate_pnl_columns then POST /pnl/recalculate
    Migration script is safe to re-run (uses ADD COLUMN IF NOT EXISTS)
  - Futu P&L history is incomplete before Apr 2026 — older trades show partial quality; sells before full history
    are excluded from totals rather than showing misleading partial numbers
  - History rebuild was failing with SQLAlchemy async errors (FIXED 2026-05-04):
    → ORM objects were being accessed after their session expired
    → Now extracts snapshot data into simple dicts while session is active
    → Flush after deletes resolves unique constraint violations on re-imports


HISTORY — WHAT WE TRIED AND WHY WE CHANGED
--------------------------------------------
  FUTU SYNC — v1: Futu OpenD local daemon
    Originally, Futu positions were synced via the Futu OpenD application — a local program that
    runs on the Mac at 127.0.0.1:11111 and exposes a Python API (futu-api package). The backend
    connected to it via routers/sync.py → services/brokers/futu_opend.py.

    Problem: OpenD only runs locally on the Mac. Railway's cloud backend at 127.0.0.1 refers to
    Railway's own container, not the user's Mac — so the website sync button always failed.
    Fix applied: the sync button showed a friendly error directing to SYNC_INSTRUCTIONS.txt.
    The workflow was: start local backend → curl /sync/futu → OpenD pulls data → writes to Railway DB.

    This worked but was cumbersome — required OpenD running, local backend running, and terminal commands.

  FUTU SYNC — v2: Gmail PDF statement (current)
    Futu HK sends a daily PDF statement to sam.sanghyunpark@gmail.com each morning. A Gmail filter
    forwards these to qkr323@gmail.com. The sync_from_email.py script reads the email, downloads the
    PDF, and uploads it to /upload/statement.

    The Futu PDF parser (services/parsers/futu.py) was rewritten to handle the daily statement format:
    - Uses text-based parsing (pdfplumber extracts each row as a separate table with no headers)
    - Two-step regex: _POS_SYM matches the symbol, _POS_DATA matches exchange/currency/price/value
    - _join_wrapped_lines() handles long ETF names that wrap across lines (e.g. TLT, EDV, KWEB, LIT)
    - Prefers "Ending Assets Overview" (end-of-day prices) over "Starting Assets Overview"

  BINANCE SYNC — geoblocked on Railway
    Binance.com is blocked on Railway's US/EU infrastructure (HTTP 451 Unavailable For Legal Reasons).
    The website sync button was returning 500 errors. Fixed with a friendly error message in SyncPanel.tsx.
    Binance must be synced from Mac terminal via local backend.
